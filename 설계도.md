좋아.
이 구조는 **“Zero-knowledge sync” (서버는 내용 모름)** 아키텍처야.

네가 적은 흐름을 실제 동작 수준까지 구체적으로 풀어보자.

---

# 🔷 전체 구조 그림

```
[기기 A]
SQLite
   ↓
AES 암호화
   ↓
HTTPS 업로드
   ↓
[서버]
암호화 blob 저장
   ↓
HTTPS 다운로드
   ↓
AES 복호화
   ↓
SQLite 반영
[기기 B]
```

핵심은:

> 🔐 서버는 메모 내용을 절대 모른다
> 🔑 키는 사용자 기기만 가진다

---

# 1️⃣ 로컬 SQLite 단계

각 기기는 로컬에 SQLite DB를 가짐.

예시 테이블:

```sql
notes(
    id TEXT PRIMARY KEY,
    content TEXT,
    updated_at INTEGER
)
```

사용자는 그냥 로컬 앱처럼 사용.

여기까지는 일반 앱과 동일.

---

# 2️⃣ AES 암호화 단계

### 🔑 키는 어디서 오나?

최초 실행 시:

* 랜덤 256bit 키 생성
* 또는 비밀번호 → KDF(PBKDF2, Argon2)로 키 생성

이 키는:

* OS 키체인에 저장
* 또는 시드 문구로 백업

---

### 🔐 무엇을 암호화하나?

두 가지 방식이 있음.

---

## 🅰 방식 A — 메모 단위 암호화

각 note를 개별 암호화

```
note.content
    ↓
AES-256-GCM
    ↓
ciphertext
```

서버에는 이렇게 저장됨:

```
{
  id: "note1",
  blob: "9af3b1c...",
  updated_at: 1712341234
}
```

✔ 동기화 충돌 관리 쉬움
✔ 일부만 수정 가능
✔ 현실적으로 추천

---

## 🅱 방식 B — DB 전체 암호화

SQLite 파일 자체를 통째로 암호화

```
database.db
    ↓
암호화
    ↓
하나의 blob
```

❌ 충돌 해결 매우 어려움
❌ 모바일 동기화에 불리

→ 실무에서는 잘 안 씀

---

# 3️⃣ 중앙 서버 단계

서버는 단순함.

예시 구조:

```sql
users(
    user_id TEXT PRIMARY KEY
)

notes(
    user_id TEXT,
    note_id TEXT,
    encrypted_blob BYTEA,
    updated_at BIGINT
)
```

서버는:

* 복호화 못함
* 키 없음
* 그냥 blob 저장

이걸 “blind storage”라고 함.

---

# 4️⃣ 다른 기기에서 동기화

기기 B가 처음 로그인하면:

1. 동일한 AES 키 확보
2. 서버에서 blob 다운로드
3. 복호화
4. 로컬 SQLite에 저장

---

# 🔥 가장 중요한 부분 — 키 공유

이게 제일 어려운 문제야.

---

## 🔐 방법 1 — 비밀번호 기반

사용자가 같은 비밀번호 입력

```
비밀번호
  ↓
Argon2
  ↓
AES Key
```

✔ 구현 쉬움
❌ 비밀번호 약하면 취약

---

## 🔐 방법 2 — 공개키 기반 (추천)

각 기기:

* 공개키 / 개인키 생성
* 서버는 공개키 저장
* 기기끼리 키 교환

이건 좀 더 복잡하지만 안전함.

---

# 5️⃣ 데이터 충돌 처리

같은 메모를 두 기기에서 수정하면?

해결책:

### 🅰 updated_at 기준 최신 승리

단순하지만 데이터 유실 가능

### 🅱 버전 필드 추가

```
version: 1,2,3...
```

### 🅲 CRDT 사용 (고급)

실시간 협업 가능
구현 난이도 매우 높음

네가 혼자 쓰는 용도면:

> updated_at 기반으로 충분
